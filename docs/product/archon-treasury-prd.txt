1. EXECUTIVE DECISION
======================================================================

Archon Treasury memakai arsitektur dua wallet.

A. AGENT RESEARCH WALLET

Jenis:
Circle Agent Wallet.

Akses:
Circle CLI melalui wrapper package.

Fungsi:
- Search service x402.
- Inspect service.
- Membayar route intelligence.
- Membayar risk report.
- Membayar fee forecast.
- Membayar compliance report bila tersedia.
- Menggunakan Gateway Nanopayments untuk microservice calls.

Saldo:
Kecil dan dibatasi.

Akses Hermes:
Hermes boleh memanggil MCP tools yang membungkus operasi research wallet.

B. TREASURY EXECUTION WALLET

Jenis:
Circle Developer-Controlled Wallet.

Akses:
Circle SDK/API dari execution worker.

Fungsi:
- Menyimpan treasury USDC.
- Melakukan bridge setelah approval.
- Menjalankan recovery.
- Menjalankan treasury transaction yang telah diotorisasi.

Saldo:
Dapat lebih besar daripada agent research wallet.

Akses Hermes:
Hermes tidak memiliki akses langsung.

Hermes hanya:
- Memahami intent.
- Memanggil research tools.
- Meminta route data.
- Membaca deterministic scoring.
- Menghasilkan recommendation.
- Membuat proposal melalui tool sempit.

Hermes tidak:
- Memegang private key.
- Memegang Circle entity secret.
- Memegang Circle API key.
- Melakukan approval.
- Memanggil DCW secara langsung.
- Membangun arbitrary calldata.
- Menjalankan bridge.
- Mengubah proposal setelah approval.

Backend deterministic menangani:
- Policy.
- Route eligibility.
- Quote validation.
- Fee calculation.
- Scoring.
- Proposal hashing.
- Approval binding.
- DCW execution.
- CCTP/App Kit bridge.
- Transaction state.
- Recovery.
- Receipt.
- Audit.

======================================================================
2. MASALAH YANG DISELESAIKAN
======================================================================

2.1 Masalah bisnis

Perusahaan atau tim crypto yang memegang USDC pada beberapa chain menghadapi
proses treasury lintas chain yang terfragmentasi.

Operator harus:
- Memeriksa saldo di beberapa chain.
- Mengetahui apakah destination chain aktif.
- Melihat biaya bridge.
- Memilih Fast atau Standard transfer.
- Memahami risiko route.
- Memeriksa recipient.
- Memastikan policy perusahaan terpenuhi.
- Melakukan transfer.
- Memantau burn, attestation, dan mint.
- Menangani transaksi yang gagal sebagian.
- Menyimpan bukti keputusan dan approval.

Bridge UI biasa membantu transaksi, tetapi bukan treasury control plane.

2.2 Masalah agent

Agent dapat membantu research dan orchestration, tetapi LLM tidak boleh menjadi
pengendali uang.

LLM dapat:
- Salah memahami amount.
- Salah membaca chain.
- Memilih quote expired.
- Retry pembayaran x402 yang sebenarnya sudah terpotong.
- Terkena prompt injection dari service response.
- Menghasilkan recipient salah.
- Mengubah parameter antara recommendation dan execution.
- Mengulang transaction saat timeout.
- Menyimpulkan transaksi berhasil hanya dari teks tool.

Archon Treasury harus memisahkan reasoning dari authorization.

2.3 Masalah pembayaran API

Agent membutuhkan data kecil secara dinamis:
- Bridge risk.
- Provider health.
- RPC health.
- Current fee signal.
- Liquidity signal.
- Sanctions screening.
- Chain status.
- Historical reliability.

Subscription API dan manual API key provisioning untuk setiap provider membuat
agent sulit memilih provider secara dinamis.

x402 dan Gateway Nanopayments memungkinkan pembelian service per request.

2.4 Masalah recovery

Bridge bukan satu atomic transaction.

CCTP/App Kit melibatkan:
- Approve.
- Burn.
- Fetch attestation.
- Mint.

Jika request timeout setelah burn berhasil, sistem tidak boleh memulai burn
baru. Sistem harus menyimpan state dan melanjutkan dari tahap terakhir.

2.5 Masalah audit

Organisasi membutuhkan jawaban untuk:
- Siapa meminta transfer.
- Instruction awal.
- Amount dan recipient yang dikonfirmasi.
- Policy version.
- Route yang tersedia.
- Data berbayar yang dibeli agent.
- Biaya x402.
- Alasan route dipilih.
- Siapa menyetujui.
- Proposal hash.
- Transaction hash.
- Amount akhir.
- Recovery yang dilakukan.

3. GOAL PROJECT
======================================================================

3.1 Product goal

Membangun control plane untuk treasury USDC lintas chain yang memungkinkan
agent melakukan research dan recommendation, tetapi uang hanya bergerak melalui
policy engine, explicit approval, dan deterministic execution.

3.2 User goal

User dapat memberi intent:

"Move 5,000 USDC from Ethereum to Arc. Maximum bridge fee 0.15%.
Prioritize safety. Do not execute without approval."

User mendapatkan:
- Normalized intent.
- Live treasury state.
- Real route estimate.
- Fast/Standard option bila didukung.
- Real paid intelligence.
- Deterministic eligibility checks.
- Deterministic score.
- Hermes explanation.
- Immutable proposal.
- Approval screen.
- Real bridge status.
- Final receipt.
- Recovery action jika terjadi partial failure.

3.3 Engineering goal

Foundation harus:
- Tidak terikat satu LLM.
- Tidak terikat satu wallet implementation.
- Tidak terikat satu bridge SDK.
- Mendukung chain/provider baru tanpa rewrite.
- Dapat dipisah menjadi service saat scale naik.
- Memiliki state machine aman.
- Tidak mengandalkan in-memory source of truth.
- Dapat di-observe dan di-audit.

3.4 Business goal

Membuktikan organisasi mau memakai agent-assisted treasury workflow karena:
- Mengurangi manual research.
- Mengurangi risiko salah route.
- Menyediakan approval dan audit.
- Menyediakan recovery.
- Memungkinkan paid intelligence per task.
- Menjaga separation of duties.

3.5 MVP success goal

Project dianggap mencapai MVP ketika:
1. User dapat membuat organization dan treasury.
2. DCW nyata dapat dibuat atau dihubungkan.
3. Agent Wallet nyata dapat dibuat atau dihubungkan.
4. Agent Wallet dapat membayar service x402 nyata.
5. Route estimate memakai data nyata.
6. Policy engine dapat menolak route nyata.
7. Hermes dapat membuat recommendation dari MCP tools.
8. Proposal immutable dapat dibuat.
9. User dapat approve lewat web.
10. DCW dapat melakukan bridge nyata menuju Arc Testnet.
11. Semua bridge step disimpan.
12. Partial failure dapat dilanjutkan.
13. Final receipt memiliki transaction hashes nyata.
14. Audit export dapat dibuat.
15. Hermes tidak dapat mengeksekusi treasury transaction.
16. Tidak ada fake data di deployed flow.

38.2 CUSTODY MODES

Archon Treasury supports adapters for multiple custody modes.

Mode A:

- Archon execution worker owns API access.
- Customer owns organization policy and approvals.
- Secrets remain in managed secret store.

CUSTOMER_MANAGED_WALLET

- Customer uses supported external wallet/custodian.
- Archon builds approved execution payload.
- Customer signer or custodian executes.
- Archon tracks status and reconciles.

- Signer runs in customer environment.
- Archon sends signed execution authorization request.
- Customer service validates proposal hash.
- Customer service signs or rejects.

READ_ONLY_TREASURY

- Archon tracks addresses.
- No execution.
- Route recommendations and proposal export only.

MVP implements Mode A.

Future architecture must use wallet adapter interfaces so Mode B-D can be added

6. State files expected to change.
7. Avoid unrelated refactor.

54.2 PR SIZE

Default:
- one coherent behavior;
- limited file count;
- reviewable diff;
- no simultaneous architecture migration and feature.

Recommended:
- under 500 changed lines excluding generated lock/migration where possible;
- one database migration per domain change;
- one primary acceptance criterion.

Large PR allowed only for:
- generated SDK;
- package extraction;
- dependency lock;
- clearly documented mechanical rename.

54.3 PROHIBITED AGENT SHORTCUTS

- fake provider.
- hardcoded success.
- TODO in financial safety path.
- arbitrary `any`.
- disabling typecheck.
- swallowing error.
- direct database access from UI.
- direct wallet access from MCP.
- using floating point for amount.
- bypassing policy for testnet.
- adding raw CLI command.
- adding generic execute transaction endpoint.
- using in-memory state for execution.
- marking completed from submitted transaction.
- automatic retry after ambiguous mutation.
- exposing stack trace to public client.
- logging secret.
- expanding tool scope without ADR/security review.

54.4 REQUIRED PR DESCRIPTION

Every PR must include:

- problem;
- scope;
- non-scope;
- architecture section affected;
- security impact;
- database impact;
- API/MCP contract impact;
- migration;
- tests;
- rollback;
- acceptance criteria.

54.5 TEST EXPECTATION

Every behavior PR:
- unit test;
- contract test if boundary changes;
- integration test if provider interaction;
- tenant isolation test if tenant data;
- idempotency test if mutation;
- failure test;
- audit assertion if financial state.

54.6 STOP CONDITION

Coding agent must stop and create an issue/ADR instead of guessing when:
- custody semantics unclear;
- provider side effect cannot be classified;
- proposal fields need material change;
- retry safety unknown;
- tenant ownership unclear;
- secret location unclear;
- chain/provider support undocumented.

62. FUTURE-FINAL ACCEPTANCE CRITERIA
======================================================================

The final platform is achieved when:

1. Multiple organizations are isolated.
2. Human users operate via web/API.
3. Customer backends operate via scoped public API.
4. Internal Hermes operates via private MCP.
5. Customer agents operate via public scoped MCP.
6. All interfaces use the same application services.
7. No agent directly executes treasury movement.
8. Multiple wallet custody modes are supported through adapters.
9. Multiple real route providers can be normalized.
10. Policy and approval are provider-independent.
11. Standing authorization, if enabled, is constrained and revocable.
12. Execution remains idempotent.
13. Recovery works per provider.
14. Reconciliation verifies final outcome.
15. Receipt is machine-verifiable.
16. Audit lineage is complete.
17. External agents cannot raise their own permissions or budget.
18. Public MCP can be disabled without affecting web/API execution.
19. Hermes can be replaced without changing financial domain.
20. A custody adapter can be replaced without changing proposal semantics.
21. A route provider can be added without changing approval engine.
22. All deployed financial integrations are real.
23. Mainnet operation follows launch gate.
24. Security zones and secrets boundaries are enforced.
25. Every future feature can be delivered through reviewable PRs.
